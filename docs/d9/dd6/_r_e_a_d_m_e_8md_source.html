3 The ORIGEN API is an Application Programmer's Interface (API) to ORIGEN
4 (Oak Ridge Isotope GENeration), a general purpose transmutation code
5 developed at Oak Ridge National Laboratory. It provides a set of classes
6 in C++/Fortran for performing depletion and decay calculations with the
7 central model equation given by the system of ODEs
8 
9 \f[ \frac{d \vec{n}}{dt} = \mathbf{A}\vec{n}(t)+\vec{F}(t) \f]
10 
11 where \f$ \vec{n} \f$ is an isotopic vector, and \f$ \mathbf{A} \f$ is a
12 transition matrix, and \f$ \vec{F} \f$ is a "feed" term.
13 The ORIGEN API can also be used to
14 - access fundamental data sources (such as decay constants, molar masses,
15  radiotoxicity factors, decay heat, and more),
16 - calculate particle emission spectra and source strengths,
17 - build hierarchical compositions, and
18 - perform unit conversions.
19 
20 See the [Changelog](CHANGELOG.md) for version information.

23 For Users {#users}
24 ================================================================================
25 
26 The ORIGEN API is distributed with [SCALE](http://scale.ornl.gov/), which may be
27 obtained through [RSICC](https://rsicc.ornl.gov/). Contact scalehelp@ornl.gov
28 for more information on obtaining the code.
29 
30 ## Building the API ##
31 
32 The ORIGEN API is callable from both C++ and Fortran. The C++ API may
33 be straightforwardly used from any SCALE installation. However, the Fortran
34 API requires Fortran module files *compiled with the Fortran compiler you will
35 build your code with*. For this reason, in order to use the Fortran API,
36 you must have access to SCALE/ORIGEN source and build SCALE alongside
37 your project.
38 
39 There are other reasons to configure and build your own ORIGEN API libraries.
40 SCALE/ORIGEN uses [CMake](https://cmake.org/) to manage
41 configuring and building on the three major platforms: Linux, Mac, and
42 Windows. With a system meeting the prerequisites, building the code is
43 as simple as `cmake $SCALE_SOURCE_DIR && make -j4 install`, assuming a Linux
44 command line. However, there are a few other `cmake` options that can be
45 handy.
46 
47 ## Configuration Details ##
48 
49 I like to make my build directories in a subdirectory *of the source directory*
50 called `build`. I typically have two builds, a release and a debug, located at
51 `$SCALE_SOURCE_DIR/build/release` and `$SCALE_SOURCE_DIR/build/debug`. I
52 then create a file called `do-config.sh` with everything I need to configure
53 for a particular build. Then I install to the subdirectory `INSTALL` within a
54 specific build directory. Here is a minimal `do-config.sh`
55 file which would build static release libraries of the ORIGEN API.
56 
57 ### Release/Static do-config.sh ###
58 
59  #Release/Static Configuration#
60 
61  #ensure that cache is blown away each time
62  rm -rf CMake*
63 
64  #example of scale data directory, source directory, and install directory
65  export SCALE_DATA_DIR=$PWD/../../data
66  export SCALE_SOURCE_DIR=$PWD/../../
67  export INSTALL_DIR=$PWD/INSTALL
68 
69  #cmake command using QUICKBUILD_ORIGEN_API
70  cmake -D CMAKE_BUILD_TYPE:STRING=RELEASE \
71  -D SCALE_DATA_DIR:FILEPATH="$SCALE_DATA_DIR" \
72  -D CMAKE_INSTALL_PREFIX:FILEPATH="$INSTALL_DIR" \
73  -D QUICKBUILD_ORIGEN_API:BOOL=ON \
74  -D BUILD_SHARED_LIBS:BOOL=OFF \
75  $SCALE_SOURCE_DIR
76 
77 
78 I do my development on Linux, and I source the
79 configuration file, `source do-config.sh && make -j4 install`. If tests are
80 enabled, then you can add a `ctest -j4` on the end to test the build.
81 
82 The main thing to notice is the `QUICKBUILD_ORIGEN_API` flag. This flag is
83 special and can only be used for the purpose of building the ORIGEN API.
84 The main additional options you may need are the following.
85  - enable/disable tests (SCALE_ENABLE_TESTS:BOOL=OFF)
86  - enable/disable Design By Contract (NEMESIS_DBC=0)
87  - control whether static or shared libraries are built (BUILD_SHARED_LIBS:BOOL=OFF)
88  - enable debugging output (DEBUG_OUTPUT=0)
89 
90 ### Debug/Shared do-config.sh ###
91 
92 To build debug libraries, I use the following config script.
93 
94  #Debug/Shared Configuration#
95 
96  #ensure that cache is blown away each time
97  rm -rf CMake*
98 
99  #example of scale data directory, source directory, and install directory
100  export SCALE_DATA_DIR=$PWD/../../data
101  export SCALE_SOURCE_DIR=$PWD/../../
102  export INSTALL_DIR=$PWD/INSTALL
103 
104  cmake -D CMAKE_BUILD_TYPE:STRING=DEBUG \
105  -D SCALE_DATA_DIR:FILEPATH="$SCALE_DATA_DIR" \
106  -D CMAKE_INSTALL_PREFIX:FILEPATH="$INSTALL_DIR" \
107  -D SCALE_ENABLE_TESTS:BOOL=ON \
108  -D NEMESIS_DBC=7 \
109  -D DEBUG_OUTPUT=0 \
110  -D BUILD_SHARED_LIBS:BOOL=ON \
111  -D QUICKBUILD_ORIGEN_API:BOOL=ON \
112  $SCALE_SOURCE_DIR
113 
114 
115 Setting `DEBUG_OUTPUT=1` can give you a lot of additional output but it
116 can be difficult figuring out where it is coming from. With tests enabled,
117 you can run them all with `ctest -j4`.
118 
119 Build time is currently about 30 cpu-minutes. So `-j4` really helps it take
120 a coffee break versus a lunch break.

122 ## Linking ##
123 
124 The ORIGEN API includes 3 libraries which you can link to.
125  - libOrigenCore: core classes (data containers and interfaces)
126  - libOrigenSolvers: CRAM and MATREX solvers
127  - libOrigenManager: additional top level methods
128 
129 If you build static libraries, then you will be able to link relatively easily.
130 The following C++ main may be used to test linking to libOrigenCore.
131 
132 ```
133 #include <iostream>
134 #include "Origen/Core/version.h"
135 int main()
136 {
137  std::cout << "This ORIGEN API is version "<<
138  Origen::version() << ".\n" << std::endl;
139  std::cout << "Here is additional configuration information:\n";
140  std::cout << Origen::config();
141  return 0;
142 }
143 ```
144 
145 Using the GNU compilers, and assuming we are in the root of the build directory,
146 the following command builds your executable.
147 
148  g++ -std=c++11 -I $INSTALL_DIR/include \
149  $INSTALL_DIR/lib/libOrigenCore.a origen_api_version.cpp
150  ./a.out
151 
152 Note that C++11 is required. If you have shared libraries, then a considerable
153 larger set of libraries is required, ordered from highest to lowest dependency
154 level in the heirarchy. The output will look something like this.
155 
156  This ORIGEN API is version 0.5.2.
157 
158  Here is additional configuration information:
159  scale_version 6.3
160  origen_api_version 0.5.2
161  build_type RELEASE
162  build_shared_libs OFF
163  release_category BETA
164  dbc_level 0
165  commit_hash 48e90437a818+
166  commit_date 2016-08-28 19:27 -0400
167 
168 For a more sophisticated executable (still static) just list all
169 the libraries.
170 
171  touch fake.txt
172  g++ -std=c++11 \
173  -L /opt/local/libexec/qt4/lib -llapack -lblas -lQtCore \
174  -I $INSTALL_DIR/include $INSTALL_DIR/lib/*.a example_new_data_type.cpp
175  ./a.out
176 
177 Note that in most cases you must supply the library path for QtCore.
178 This main demonstrates how to add a new ORIGEN data type.
179 
180 ```
181  #include <memory>
182  #include <sstream>
183 
184  #include "Origen/Core/io/ReactionResourceIO.h"
185  #include "Origen/Core/version.h" /*version(), config()*/
186  #include "ScaleUtils/IO/Binary.h"
187 
188  using namespace Origen;
189 
190  namespace user
191  {
192 
193  // inherit from the binary format reader
194  // and implement the methods to read/write your format from a stream
195  class MyIO_xyz : public ScaleUtils::IO::Binary<ReactionResource>
196  {
197  public:
198  // read an object from an istream
199  bool read( ReactionResource& obj,
200  std::istream& stream,
201  ScaleUtils::IO::DB opts )
202  {
203  // insert actual code here and return false if failed
204  return true;
205  }
206 
207  // write an object to an ostream
208  bool write( const ReactionResource& obj,
209  std::ostream& stream,
210  ScaleUtils::IO::DB opts )
211  {
212  // insert actual code here and return false if failed
213  return true;
214  }
215  };
216  }
217 
218  int main()
219  {
220 
221  // either add as a default format (valid for any call anywhere)
222  ReactionResourceIO::default_add(
223  "xyz", ReactionResourceIO::SP_Format( new user::MyIO_xyz() ) );
224  ReactionResourceIO rr_io;
225  // or just add your formats to this specific instance
226  // rr_io.add("xyz",SP_Format( new user::MyIO_xyz() ) );
227 
228  ReactionResource rr;
229 
230  ScaleUtils::IO::DB opts;
231  rr_io.load( rr, "fake.txt", opts );
232  std::cout << opts << std::endl;
233 
234  return 0;
235  }
236 ```

249 For Developers {#developers}
250 ================================================================================
251 
252 The ORIGEN API is organized into 3 main subpackages for the
253 %Core, %Solver, %Manager. The backbone of the API is written in C++,
254 with special Fortran bindings developed for the most important features.
255 One is produced for each subpackage.
256 If Fortran is enabled, the Fortran libraries have an `_f` appended at the
257 end.
258 
259 Each subpackage directory is further organized into subdirectories.
260 
261 - [Core](@ref Core)
262  - [dc](@ref Core/dc) - data containers
263  - [ut](@ref Core/ut) - utilities
264  - [io](@ref Core/io) - I/O routines
265  - [ts](@ref Core/ts) - transition system
266  - [xf](@ref Core/xf) - interfaces
267  - [re](@ref Core/re) - resources
268  - [fn](@ref Core/fn) - functions
269 - [Solver](@ref Solver)
270  - [SolverSelector](@ref Solver/SolverSelector.h) - choose solver
271  - [cram](@ref Solver/cram) - CRAM solver
272  - [matrex](@ref Solver/matrex) - original ORIGEN solver
273 - [Manager](@ref Manager)
274  - [emit](@ref Manager/emit) - fortran particle emission
275  - [wrap](@ref Manager/wrap) - fortran wrappers
276  - [libld](@ref Manager/libld) - library building (experimental)
277 
278 Scattered throughout are subdirectories called "f", "c", "tests", and
279 "examples" which contain CIX Fortran binding files, C interface
280 files, tests, and examples, respectively.
